# Hush 钉钉消息新鲜度修复设计

## 问题与证据

Hush 会话列表会按 `received_at` 倒序排列，但当前本地收件箱的最新钉钉消息停在 2026-07-17 13:27。DWS 在 2026-07-20 的只读查询能返回当天新消息，说明钉钉数据源正常，问题位于 HUMHUM 的同步路径。

根因有三处：

1. 顶部“刷新”按钮只重新读取本地收件箱，没有调用 DWS 同步。
2. 自动同步默认关闭；即使用户已经明确开启，应用启动后也要先等待一个完整的五分钟周期才会同步。
3. 会话列表永远只显示时分，使数天前的消息看起来像当天消息。

## 已选方案

采用用户确认的方案 A，保留显式授权边界：

- DWS 自动同步仍由用户明确开启，不因发现登录态而偷偷启用。
- 顶部“刷新”在 DWS 已认证时执行一次真实的只读同步，再刷新本地收件箱；未认证时仍刷新本地来源并显示连接提示。
- 自动同步已开启时，应用启动后立即执行一次同步，之后每五分钟同步一次。
- 当前设备在本次安装验证中明确开启自动同步。
- 会话时间按相对日期显示：今天显示 `HH:mm`，昨天显示 `昨天 HH:mm`，本年更早显示 `M月D日`，跨年显示 `YYYY/M/D`。

## 数据流

1. 用户点击顶部刷新。
2. 前端检查 DWS 状态。
3. 已认证时调用 `sync_hush_dws`；同步完成后并行刷新 DWS 状态和本地收件箱。
4. 同步失败时保留现有消息，并在 Hush 连接状态区域显示可读错误。
5. 后台任务仅在持久化配置 `auto_sync_enabled=true` 时运行；启动同步完成后进入固定周期。

## UI 行为

- 顶部刷新按钮在真实同步期间显示旋转状态并禁用，避免重复触发。
- 按钮提示明确为“同步并刷新钉钉消息”，不再暗示只是重绘界面。
- 非当天消息必须显示日期，完整本地时间保留在 `<time>` 的提示文本中。
- 排序规则不变：特别关注优先级之外，同一分组内仍按最后一条消息的真实时间倒序。

## 错误与隐私

- 所有 DWS 操作保持只读，不发送或回复消息。
- 未登录、令牌过期或同步失败不清空本地收件箱。
- 自动同步只在用户明确开启后持久化，不把“已登录”当作后台读取授权。

## 测试与验收

- 前端回归测试证明顶部刷新会调用真实同步，而不是只调用 `get_hush_inbox`。
- 时间格式测试覆盖今天、昨天、本年更早、跨年和无效时间。
- Rust 测试证明自动同步开启时启动即触发，关闭时不触发。
- 运行 Hush/DWS 定向测试、完整前端测试、Rust 测试和生产构建。
- 安装新应用后确认当前设备自动同步已开启，点击刷新后本地最新时间达到 DWS 可见的新消息时间。
