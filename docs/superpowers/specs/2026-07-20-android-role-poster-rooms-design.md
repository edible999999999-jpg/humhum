# HUMHUM Android 角色海报空间设计

## 状态

用户于 2026-07-20 选定 HTML 原型的第二套方向：**角色海报**。

本规范取代 `2026-07-19-android-room-background-navigation-design.md` 中偏弱的
“低透明度全页背景”表现，但不改变该规范和
`2026-07-19-android-companion-rooms-design.md` 已确立的数据、隐私、权限与角色边界。

原型：

- `.stitch/designs/android-quiet-rooms/index.html`
- 选定概念：`B / 角色海报`
- 设计系统：`.stitch/DESIGN.md`

## 目标

让 Android 成为 Mac HUMHUM 的移动伴侣，而不是另一套重新发明的产品：

- 四个角色使用 Mac 当前的原始角色背景，不重新生成或重新设计角色。
- 角色图承担空间识别和情绪，不承担导航、按钮或信息表达。
- 每个房间按自己的工作职责组织内容，不复用同一个“标题 + 摘要 + 卡片列表”模板。
- Humi 保留 Android 独有的伴侣入口；Hype、Hush、Hexa 的职责与 Mac 一致。
- 每条个人信息都能回答“从哪里来、是否已授权、什么时候更新、是否可执行”。

## 核心设计判断

### 角色是环境，不是贴图

每个角色的原始背景占据首屏上部的一段连续空间，并按角色构图裁切。角色可以被边缘裁切，
但不能被缩成独立小图、头像、卡片插画或 Tab 图标。

内容可以直接进入背景原本的留白区域。不得为了可读性在角色上覆盖大面积磨砂卡片；
应通过裁切、文字位置和图片透明度保证可读性。

### 海报不是营销 Hero

角色海报只回答：

1. 当前是谁；
2. 当前最值得用户知道什么；
3. 下一步从哪里开始。

它不展示产品介绍、功能说明、宣传口号或装饰性数据。首屏仍需露出主操作或下一段真实内容。

### 四个房间必须结构不同

- Humi 以对话和解释为中心。
- Hype 以搜索、分类和复用为中心。
- Hush 以人、关系和待处理消息为中心。
- Hexa 以用户确认、Agent 当前步骤和监督为中心。

共享的是品牌、字体、导航、间距和信任语言，不是页面模板。

## 不可变角色资产

| 角色 | 唯一背景资产 | 移动构图原则 |
| --- | --- | --- |
| Humi | `public/mascots/hub-backgrounds/humi-room.webp` | 居中偏上，保留闭眼与耳机，文字避开面部中心 |
| Hype | `public/mascots/hub-backgrounds/hype-room.webp` | 依据原图主体方向保留知识角色完整表情 |
| Hush | `public/mascots/hub-backgrounds/hush-room.webp` | 优先保留原图中 Hush 的真实轮廓与窥视构图，不得用 Humi/Hype 替代 |
| Hexa | `public/mascots/hub-backgrounds/hexa-room-v2.png` | 保留黄色 Hexa 与工具身份，不得使用其他水母换色替代 |

这些图片是装饰层：

- 无可访问性标签；
- 不拦截触摸；
- 不进入底部导航；
- 不由生成模型重绘；
- 不从 GLB 截图临时替代；
- 不增加白边、方形底或独立阴影。

## 共享外壳

### 顶部

- Android 系统状态区之下使用 60dp 工具栏。
- 左侧显示当前角色和一个短职责标签。
- 右侧最多显示同步状态与设置图标。
- 同步状态使用普通文字和小状态点，不使用大胶囊。

### 底部导航

四个固定目的地与 Mac 语义一致：

| 角色 | 图标 |
| --- | --- |
| Humi | `Mic2` |
| Hype | `RadioTower` |
| Hush | `Eye` |
| Hexa | `Wrench` |

导航使用图标加短标签、2dp 选中标记和稳定的 68dp 高度。禁止角色头像、填充式选中胶囊、
动态改变目的地尺寸或重新生成一套 Tab 背景。

### 字体与几何

- 中文正文：Noto Sans SC。
- Latin 角色名与数字：Inter 或等宽数字特性可用的系统无衬线字体。
- 页面重点句：22-26sp，最多三行。
- 区域标题：17sp。
- 正文：14-15sp。
- 来源与状态：12sp。
- 字距恒为 0。
- 卡片、按钮和输入框最大 8dp 圆角。
- 触摸目标至少 48dp。
- 页面区域保持连续；卡片只用于一条独立对话、消息、审批或可复用知识对象。

## 六个页面

### 1. 配对页

目标：让用户安全地把 Android 连接到自己的 Mac。

结构：

1. Humi 品牌海报区，显示 `HUMHUM` 与“连接这台电脑”。
2. 主操作：扫描 Mac 中 Hexa 显示的配对二维码。
3. 次操作：粘贴配对资料。
4. 当前连接状态或错误。
5. 手动地址、配对码和证书指纹收在“连接遇到问题”披露区。
6. 底部说明资料保存在 Android 私有存储并受 Keystore 保护。

配对前不出现四角色底部导航。

### 2. Humi

目标：让用户先与 Humi 说话，再自然带入今天、记忆和身体信号。

第一屏：

1. 约 230-250dp 的 Humi 环境海报。
2. 一句基于已确认上下文的判断，例如“先把 Android UI 重构这一件事放稳”。
3. 与海报连续衔接的对话输入，含语音与发送图标。
4. 下一段“今天”内容露出一部分。

后续：

- 今天：明确、未完成的目标或用户选择的任务。
- 我记得：仅确认过的记忆、偏好和习惯。
- 身体信号：步数、静息心率、睡眠的日汇总。
- Humi 建议：与事实分开，必须由用户确认后才能写入今天或长期记忆。

Humi 是唯一可以用伴侣式语气整合跨域信息的角色，但不得把推测说成事实。

### 3. Hype

目标：让知识和技能在下一次工作中直接复用。

第一屏：

1. Hype 环境海报，展示本次知识库最值得处理的一个主题。
2. 搜索框固定为该页面最明显的操作。
3. 首个知识分组露出，不用统计卡片占据首屏。

主要分组：

- 可复用 Skill；
- 已确认偏好；
- 已确认习惯模板；
- 长期记忆；
- 用户授权的 Obsidian 摘要。

列表按“能做什么”组织，而不是按文件夹、扩展名和绝对路径组织。每行显示标题、用途摘要、
来源类别和更新时间。

### 4. Hush

目标：从用户视角看清谁需要关注，以及怎样回复才合适。

第一屏：

1. Hush 原始环境海报。
2. 一句注意力判断，例如“有两个人值得你今天回一下”。
3. 只读与授权状态紧邻该判断。
4. 第一条需要关注的联系人或会话露出。

主要内容：

- 需要回复；
- 特别关注；
- 家人、朋友、工作和其他已授权来源；
- 关系上下文；
- 建议语气；
- 用户主动打开后的有限消息预览。

禁止：

- 自动回复；
- 在没有授权时暗示消息已连接；
- 展示完整原始聊天数据库；
- 把健康数据放进 Hush；
- 使用错误角色图或通用水母替代 Hush。

### 5. Hexa

目标：让用户先处理需要自己决定的事，再观察 Agent 进展。

第一屏：

1. Hexa 原始黄色角色海报。
2. 当前权限：只读观察或可以控制。
3. 需要用户确认的动作数量与最紧急的一条动作。
4. `允许` / `拒绝` 只在控制权限存在时显示。

后续：

- 正在关注的目标；
- Agent 当前步骤；
- 阻塞与漂移；
- 最近完成结果；
- 用户打开后查看的有限对话；
- 可追问 Agent 的输入区。

Hexa 不展示多 Agent 拓扑图，也不成为新的编排平台。

### 6. 设置

设置页不使用角色海报。它是安静、可扫描的工具列表：

- Mac 连接；
- 健康数据权限与来源；
- 后台健康同步；
- Hexa 后台监控；
- 电池与自启动；
- 本机加密；
- 删除 Android 本地数据；
- 版本信息；
- 默认折叠的高级诊断。

删除操作使用风险色，其余设置不借用角色主色制造误导。

## 数据来源与可见状态

| 页面内容 | 来源 | 移动端展示规则 |
| --- | --- | --- |
| Humi 今天 | Mac `HexaGoalStore`、`HexaWatchStore`、用户选择的 Obsidian 任务 | 只展示明确且未完成的条目 |
| Humi 记忆与偏好 | Mac `KnowledgeStore` | 只展示用户保存或确认的摘要 |
| Humi 习惯 | 结构化 habit store | 建议与已确认习惯分开 |
| 身体信号 | Android Health Connect；步数可回退到本机计步器 | 仅日汇总，逐项显示权限、来源与新鲜度 |
| Hype 知识 | `KnowledgeStore`、skill index、授权的 Obsidian index | 标题、人话摘要、类型、归属、更新时间；无原始路径 |
| Hush 消息 | `HushStore` 与用户明确授权的只读桥 | 有界摘要、未读/关注状态、来源平台；无自动发送 |
| Hexa | `HexaGoalStore`、`HexaWatchStore`、移动会话投影 | 保留 read/control 权限边界与有限对话 |

离线时继续展示最后一份有效加密快照和更新时间，但关闭需要 Mac 在线的动作。过期建议不可继续执行。

## 页面状态

每个房间独立支持：

- 有数据；
- 空状态；
- 未授权；
- 同步中；
- 离线缓存；
- 数据过期；
- 局部来源不可用；
- 操作失败。

空状态必须说明缺少的是哪种来源或授权，不能统一写“暂无数据”。

## 动效

- 房间切换使用短交叉淡入。
- 海报图固定，不做视差和持续漂浮。
- 只在配对、刷新、批准、拒绝、发送时显示局部进度。
- 尊重系统减少动态效果设置。

## 实现边界

- 设计落地目标是 Jetpack Compose，不把 HTML 运行时代码移植进应用。
- 复用现有 `HumHumUiState`、`MobilePersonalContext` 与 callback 边界。
- UI 不直接访问网络、Health Connect 或持久化层。
- 优先重构共享外壳和页面信息架构；不在这次视觉改造中扩展新的隐私权限或网络协议。

## 验收条件

1. 390x844 第一屏能同时看见角色环境、当前判断和主操作或下一段内容。
2. 360x800 与 412x915 无水平滚动、截字和控件重叠。
3. Android 字体缩放 1.0 与 1.3 均保持可读。
4. 四角色使用指定 Mac 原始背景，Hush 与 Hexa 不得错图。
5. 底部导航只使用 `Mic2`、`RadioTower`、`Eye`、`Wrench` 的等价图标和文字标签。
6. Humi 对话、Hype 搜索、Hush 联系人注意力、Hexa 用户确认分别是各页第一操作。
7. 所有个人数据均有来源、授权和新鲜度表达。
8. Hush 不自动回复，Hexa 不在只读权限下显示控制操作。
9. 设置页不使用角色海报。
10. 现有配对、健康同步、离线快照、会话控制、撤销设备权限相关测试继续通过。
